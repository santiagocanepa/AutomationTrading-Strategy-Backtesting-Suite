## Validation Checks

Each partition is validated before storage using `DataValidator`:

| Check               | Description                                          | Severity |
|---------------------|------------------------------------------------------|----------|
| Column presence     | `open`, `high`, `low`, `close`, `volume` required    | error    |
| Monotonic index     | Timestamps strictly increasing                       | error    |
| OHLC sanity         | `high >= max(open,close)`, `low <= min(open,close)`  | error    |
| Non-negative volume | `volume >= 0`                                        | error    |
| Gap detection       | Flag gaps > 2× expected interval                     | warning  |
| Duplicate detection | No duplicate timestamps                              | error    |

## Storage Format

- **Format**: Apache Parquet with ZSTD compression (level 3)
- **Partitioning**:
  - Intraday (1m–4h): Monthly partitions (`YYYY-MM.parquet`)
  - Daily+ (1d, 1w, 1M): Yearly partitions (`YYYY.parquet`)
- **Metadata**: Each partition stores custom Arrow metadata:
  - `exchange`, `symbol`, `timeframe`, `source`, `date_min`, `date_max`

## Methodology Notes

1. **Source priority**: BinanceVision for historical months, CCXT for current month
2. **Incremental sync**: Only missing partitions are downloaded on re-run
3. **CCXT symbol normalization**: `BTCUSDT` → `BTC/USDT` for CCXT API calls
4. **Timestamp normalization**: Binance Vision archives are parsed with automatic epoch-unit detection (`ms`/`us`/`ns`)
5. **Source tracking**: Each partition records whether data came from `binance_vision` or `ccxt_binance`
